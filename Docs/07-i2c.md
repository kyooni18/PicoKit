# PicoKit Documentation

## I2C and SPI


Choose the I2C controller, bus speed, and the two pins up front:

```swift
let i2c = try PicoI2C(
    .i2c0,
    frequency: .kilohertz(400),
    sda: try PicoPin(4),
    scl: try PicoPin(5)
)
```

`i2c.actualFrequency` reports the bus rate actually selected by the SDK after
its divider quantization.

Writing a small command or register payload looks like this:

```swift
let count = try i2c.write(
    address: 0x3C,
    bytes: [0x00, 0xAF],
    timeout: .milliseconds(100)
)
```

Reading returns exactly the requested number of bytes or throws:

```swift
let bytes = try i2c.read(
    address: 0x3C,
    count: 4,
    timeout: .milliseconds(100)
)
```

PicoKit accepts normal 7-bit peripheral addresses from `0x08` through `0x77`.
Reserved addresses outside that range are rejected before the SDK sees them.
SDA and SCL must be different physical pins.

An empty write is a validated no-op, and `read(count: 0, ...)` returns an
empty array without touching the bus. This avoids the Pico SDK's assertion on
zero-length I2C transactions.

Each call is a complete standalone read or write transaction with STOP enabled
by default. If a sensor needs a repeated-start register transaction, suppress
the write STOP and issue the read next:

```swift
try i2c.write(
    address: 0x3C,
    bytes: [0x10],
    timeout: .milliseconds(100),
    stop: false
)
let bytes = try i2c.read(address: 0x3C, count: 4, timeout: .milliseconds(100))
```

For the common register-prefix form, `writeRead` performs that sequence for
you. It validates the address, read count, and timeout before issuing either
transaction; its timeout applies independently to the write and read portions:

```swift
let bytes = try i2c.writeRead(
    address: 0x3C,
    bytes: [0x10],
    count: 4,
    timeout: .milliseconds(100)
)
```

Reads issue STOP by default as well. Pass `stop: false` when the following
operation must continue with a repeated START, such as a read-then-write
command sequence.

### SPI

Choose SPI0 or SPI1 with its pins and clock explicitly:

```swift
let spi = try PicoSPI(
    .spi0,
    frequency: .megahertz(8),
    sck: try PicoPin(18),
    mosi: try PicoPin(19),
    miso: try PicoPin(16)
)
let received = try spi.transfer([0x9F, 0, 0, 0], timeout: .milliseconds(100))
```

SPI is full duplex: the returned array matches the transmitted length.
Pass `chipSelect` to let `PicoSPI` configure and drive the conventional
active-low chip-select (high while idle) with `select()` and `deselect()`. If it is `nil`, drive chip select
with another GPIO around each transaction. When `chipSelect` is supplied without
an explicit `gpio`, firmware selects a GPIO controller for its compiled RP2040
or RP2350 target. The initializer exposes SPI mode,
bit order, and 8/16-bit frame width; both `transfer([UInt8], timeout:)` and
`transfer([UInt16], timeout:)` are available for full-duplex configurations
with MISO. For a receive-only device, `read(count:repeatedByte:)` clocks 8-bit
frames while transmitting the selected filler byte on MOSI. In 16-bit mode,
`read(_:repeatedWord:)` provides the equivalent repeated-word operation. SCK, MOSI, and MISO must be different physical pins, and an
explicit chip-select must differ from all three.
Use the `timeout:` overloads of either receive-only method when a stalled
peripheral must be bounded; the overloads without a timeout remain blocking.
The 16-bit write overload also accepts `timeout:` and reports a partial
transfer if the peripheral stops making progress.

### DMA output

For a large, already-prepared display or device buffer, use the synchronous DMA
write overload that matches the configured frame width:

```swift
try spi.writeDMA(frameBytes)       // SPI configured for 8-bit frames
try spi.writeDMA(frameWords)       // SPI configured for 16-bit frames
let receivedBytes = try spi.transferDMA(frameBytes) // full-duplex 8-bit DMA
let receivedWords = try spi.transferDMA(frameWords) // full-duplex 16-bit DMA

try spi.writeDMA(frameBytes, timeout: .milliseconds(100))
let bounded = try spi.transferDMA(frameBytes, timeout: .milliseconds(100))
```

The calls claim the required paired DMA channels and wait for transfer
completion. `transferDMA` requires MISO and returns the frames received while
the transmit buffer is clocked. `writeDMA` discards received frames while
keeping the SPI RX FIFO drained.
PicoKit retains those channels so repeated writes can reuse them; call
`spi.releaseDMAChannels()` when the SPI peripheral is no longer using DMA.
The SPI object also releases its retained channels during deinitialization.
The Swift array is never retained after the call returns. The timeout overloads
abort both retained channels before returning; use them when a stalled SPI
peripheral must not block forever.

`PicoUART.writeDMA(_:)` follows the same synchronous-buffer rule for UART TX;
call `uart.releaseDMAChannel()` when the UART no longer needs DMA, or let the
UART object deinitialize for automatic cleanup.
DMA reduces per-byte CPU work, but total wall time can still be limited by the
configured SPI clock or UART baud rate.

# v0.2 application interoperability and migration

PicoKit 0.2 keeps the library focused on hardware abstraction while allowing a
firmware project to own its C boundary. The internal SDK bridge remains limited
to Pico SDK access and fixed-width C-compatible values. Application adapters,
third-party headers, module maps, and callbacks live entirely in the generated
project.

## SPI for displays

```swift
let spi = try PicoSPI(
    .spi0,
    frequency: .megahertz(40),
    sck: .gpio18,
    mosi: .gpio19,
    miso: nil,
    mode: .mode0,
    chipSelect: .gpio17
)

try spi.select()
try spi.write(pixelBytes)
try spi.deselect()
print(spi.actualFrequency.hertz)
```

Write-only bulk operations call the Pico SDK blocking transfer directly and do
not allocate an RX buffer. The timeout overload reports a partial transfer.
Choose `.sixteen` data bits at initialization before calling the `[UInt16]`
write overload. General DMA is intentionally outside the 0.2 API.

## GPIO setup

Use `configure` when the initial level matters:

```swift
let gpio = PicoGPIO(chip: .rp2350)
try gpio.configure(
    .gpio17,
    mode: .output,
    initialState: .high,
    pull: .none,
    driveStrength: .milliamps8,
    slewRate: .fast
)
```

The bridge writes the output latch before enabling output direction, preventing
a low pulse on active-low chip-select and reset pins. `resetPulse` and
`PicoBacklight` provide the common LCD reset and PWM-backlight operations.

## Migrating 0.1 projects

Existing initializers, `transfer(_:timeout:)`, Arduino-style GPIO calls, and
`Firmware/Dependencies.cmake` continue to build. Migration is explicit:

```sh
swiftpico dependencies migrate
swiftpico dependencies resolve
swiftpico build
```

Commit `Package.resolved`, `Firmware/dependencies.json`,
`Firmware/dependencies.lock`, and `Firmware/Generated/Dependencies.cmake`.
Never commit `Firmware/Dependencies.local.cmake`.
